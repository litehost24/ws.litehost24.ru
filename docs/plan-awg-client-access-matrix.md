# Матрица правил: подписка, устройство, привязка, импорт

Актуально на 12 апреля 2026 года.

Этот документ дополняет:
- `docs/plan-awg-client-multiplatform.md`
- `docs/plan-awg-client-backend-phase1.md`

Он фиксирует правила для новой модели без login flow в приложении.

## Главное решение

В приложении больше нет разделения на:
- owner app user
- delegate app user

С точки зрения приложения есть только два типа профиля:
- `ManagedProfile`
- `ImportedProfile`

Кто именно пользуется телефоном, для приложения неважно:
- владелец подписки;
- мама;
- другой получатель.

Если устройство привязано к подписке через invite, оно выглядит одинаково.

## Термины

### `SiteOwner`

Пользователь сайта Laravel, который:
- покупает подписку;
- управляет ею в кабинете;
- создаёт invite;
- отзывает и перевыдаёт доступ.

### `UserSubscription`

Одна карточка подписки на сайте.

### `AppDevice`

Конкретная установка приложения на телефоне или компьютере.

### `ManagedProfile`

Профиль приложения, созданный после успешной привязки конкретной `UserSubscription`.

### `ImportedProfile`

Локальный профиль, созданный из вручную импортированного `AmneziaWG` конфига.

### `SubscriptionInvite`

Короткоживущий одноразовый токен для привязки одной `UserSubscription` к одному `AppDevice`.

### `SubscriptionAccess`

Серверная запись, что конкретная подписка сейчас выдана конкретному устройству.

## MVP-модель

### Кардинальности

- Один `SiteOwner` имеет много `UserSubscription`.
- Одна `UserSubscription` в MVP имеет максимум одно активное managed-устройство.
- Один `AppDevice` в MVP содержит один текущий профиль:
  - либо `ManagedProfile`;
  - либо `ImportedProfile`.

### Что это значит practically

- если владелец купил отдельную подписку для Мамы, это отдельная карточка на сайте и отдельная
  привязка на её телефон;
- если владелец хочет использовать ещё одну подписку на своём втором телефоне, для неё нужна
  отдельная карточка подписки или отдельная продуктовая политика;
- приложение в первой версии не показывает одновременно несколько managed-профилей.

## Роли и полномочия

### `SiteOwner`

Имеет право:
- купить и продлить подписку;
- открыть карточку подписки;
- сгенерировать invite;
- увидеть, на каком устройстве подписка активирована;
- отозвать устройство;
- перевыдать привязку;
- видеть статус подписки и срок.

Не делает в приложении:
- login;
- просмотр всего списка подписок аккаунта.

### `Managed app user`

Это любой человек, у которого на телефоне активирована привязка по invite.

Имеет право:
- активировать invite в приложении;
- видеть статус одной привязанной подписки;
- подключать и отключать VPN;
- получать автообновление конфига;
- видеть простую ошибку, если доступ отозван или подписка истекла.

Не имеет права:
- видеть кабинет сайта;
- видеть другие подписки;
- продлевать, покупать или оплачивать;
- генерировать новые invite.

### `Imported app user`

Пользователь ручного режима.

Имеет право:
- импортировать локальный конфиг;
- подключать и отключать VPN;
- заменить конфиг вручную.

Не имеет права:
- рассчитывать на автообновление от сервера;
- видеть серверный статус подписки;
- использовать invite API без перепривязки режима.

## Матрица возможностей

| Действие | SiteOwner | Managed app user | Imported app user |
| --- | --- | --- | --- |
| Купить/продлить подписку на сайте | Да | Нет | Нет |
| Создать invite | Да | Нет | Нет |
| Отозвать устройство | Да | Нет | Нет |
| Активировать привязку в приложении | Нет | Да | Нет |
| Получать актуальный managed-конфиг | Нет | Да | Нет |
| Подключать/отключать VPN | Нет | Да | Да |
| Импортировать локальный конфиг | Нет | Нет | Да |
| Автообновлять конфиг от WS backend | Нет | Да | Нет |

## Матрица сценариев

| Сценарий | Кто начинает | Что делает система |
| --- | --- | --- |
| Владелец привязывает подписку себе | SiteOwner | На сайте генерирует invite, открывает его на своём телефоне |
| Владелец привязывает подписку Маме | SiteOwner | На сайте генерирует invite и передаёт ссылку/QR/код |
| Мама активирует invite | Managed app user | Приложение получает managed profile только для одной подписки |
| Владелец перевыдаёт подписку | SiteOwner | Старое устройство отзывается, новая привязка становится единственной активной |
| Телефон потерян | SiteOwner | Отзывает старое устройство и выдаёт новый invite |
| Пользователь пришёл со сторонним конфигом | Imported app user | Приложение работает без backend и без server sync |
| Сервер у подписки сменился | System | Только managed profile получает новый конфиг по `config_version` |

## Правила по устройствам

### Правило 1

Одна подписка в MVP привязывается максимум к одному активному managed-устройству.

Это главный инвариант первой версии.

### Правило 2

Одно приложение в MVP работает только с одним текущим профилем.

То есть:
- либо managed binding;
- либо imported config.

### Правило 3

Перепривязка или ручной импорт, если профиль уже существует, требуют явного подтверждения и
заменяют текущий профиль.

### Правило 4

Одновременно активен только один VPN-туннель.

## Правила по привязке

### Invite

Invite:
- одноразовый;
- короткоживущий;
- привязан к одной `UserSubscription`;
- после активации сгорает;
- может быть отозван до активации.

Форматы:
- ссылка;
- QR;
- короткий код.

### Результат привязки

После успешной активации invite приложение получает:
- device session;
- managed profile;
- право запрашивать `manifest`;
- право запрашивать `config`.

На сервере в этот момент должно происходить не только создание access-записи, но и техническая
перепривязка:
- старый peer / старый клиентский конфиг инвалидируется;
- для нового устройства выпускается новый peer / новый конфиг.

### Повторная выдача

Повторная выдача invite:
- отзывает старую привязку;
- отзывает старую device session;
- делает новое устройство единственным активным.

## Правила по конфигу

### Для `ManagedProfile`

- конфиг приходит с backend;
- backend хранит `config_version`;
- приложение регулярно спрашивает `manifest`;
- новый конфиг применяется автоматически.
- при новой привязке старый managed-конфиг должен переставать работать технически, а не только
  административно.

### Для `ImportedProfile`

- конфиг хранится только локально;
- backend ничего не знает об этом профиле;
- обновление только вручную;
- никакого `manifest` нет.

## Локальные настройки маршрутизации

Для обоих типов профиля Android V1 должен поддерживать локальный split tunneling:
- `off`
- `include_selected_apps`
- `exclude_selected_apps`

А также локальный список выбранных пакетов приложений.

Эти настройки:
- не хранятся на backend;
- не входят в invite;
- не входят в server-managed конфиг;
- не должны теряться при автообновлении managed-конфига.

## Правила по отображению в приложении

### Если профиль `ManagedProfile`

Приложение показывает:
- название привязанной подписки;
- статус;
- срок;
- кнопку подключения;
- статус синхронизации конфига;
- сообщение `доступ отозван` / `подписка неактивна`, если нужно.

### Если профиль `ImportedProfile`

Приложение показывает:
- локальное имя профиля;
- кнопки подключения и замены конфига;
- пометку, что это пользовательский конфиг и он обновляется вручную.

### Что приложение не показывает в MVP

- общий кабинет;
- список всех подписок аккаунта;
- пополнение;
- оплату;
- переходы на сайт;
- сложный multi-profile интерфейс.

## Recovery и ошибки

| Ситуация | Что видит пользователь | Что делает система |
| --- | --- | --- |
| Invite истёк | `Ссылка устарела` | Нужен новый invite |
| Привязка отозвана | `Доступ отключён` | Session больше не работает |
| Подписка истекла | `Подписка неактивна` | Config не выдаётся до продления |
| Телефон потерян | Старое устройство перестаёт работать | Владелец выдаёт новый invite |
| Конфиг не обновился | `Не удалось синхронизировать` | Текущий рабочий конфиг сохраняется до успешной замены |
| Пользователь хочет сторонний сервер | Выбирает `Импортировать конфиг` | Backend WS VPN не участвует |

## Что считаем правильным для MVP

1. Убираем login flow из приложения.
2. Основной режим приложения: `Привязать подписку`.
3. Второй режим: `Импортировать конфиг`.
4. Одна подписка = одно активное managed-устройство.
5. Один install = один текущий профиль.
6. Только managed-режим получает автообновление конфига.
7. Сайт управляет привязкой, но приложение на сайт не ведёт.
8. Новая привязка всегда должна технически перевыпускать peer/config.
9. Split tunneling — локальная настройка Android-клиента, не связанная с server sync.

## Что откладываем

- несколько managed-профилей на одном устройстве;
- список всех подписок аккаунта внутри приложения;
- отдельный login владельца в приложении;
- общий cloud-аккаунт внутри приложения;
- автоматическое обновление импортированных сторонних конфигов.

## Итог

Новая модель проще старой:
- сайт управляет подпиской;
- invite привязывает подписку к устройству;
- приложение использует одну подписку как managed profile;
- сторонние пользователи остаются возможны через imported mode;
- Play-safe путь не зависит от in-app переходов на сайт.
